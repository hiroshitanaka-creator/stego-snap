# stego-snap マイルストーン（v1.0までの全体像）

## 0. 北極星（North Star）
「人間には見えて、AI視覚モデルには見えにくい」表現を含む、**完全クライアントサイド**のステガノグラフィ・トイボックスを提供する。  
ただし “AIに勝つ” ではなく、**プライバシーを成立させる**ことが目的。

## 1. 非交渉の制約（Non-negotiables）
- 100% Client-Side（サーバー送信なし）
- No Tracking（Analytics/Cookie/Beacon等の導入禁止）
- GitHub Pagesでホスト可能（静的配布）
- OSSとして検証可能（難読化で逃げない）

## 2. 脅威モデル（暫定）
### 守りたいもの
- 第三者が“画像を見ただけ”で内容を読み取れないこと
- AI視覚モデルが“推測”で当てにくいこと（ただしコード実行可能な相手には原理上読める/解ける可能性がある）

### 想定しない（非目標）
- 国家レベル攻撃者への耐性
- 画像がSNSで再圧縮された後もLSBが残ること（LSBは壊れる前提）
- 暗号の自作（暗号は標準アルゴリズムを使う）

## 3. “完成(DoD)”の定義（v1.0）
v1.0 の合格条件：
- 各ツールが **Encode/Decodeの往復（roundtrip）** を規定条件で達成
- 既知の壊れ方（SNS再圧縮/リサイズ等）をドキュメントで明示
- 共通の制約/用途/限界がREADMEまたはdocsに明文化
- 主要ブラウザ（最低：Chrome / iOS Safari）で致命的エラー無し
- “未実装の主張”をREADMEに残さない（実装と説明が一致）

## 4. 現在の構成（As-Is）
- `index.html`：Stego Snap（LSB）
- `noise.html`：Noise Talk（グリッド）
- `prism.html`：Prism Code（色タイル）
- `multi-layer-stego.html`：Multi-Layer Stego（複合＋AES）
- `animated-prism.html`：Animated Prism（※状態要確認。未実装要素があればv1.0対象外にするかMVP化する）
- `README.md`, `LICENSE`

## 5. 既知の問題（Known Issues）
- UIイベント依存・チャンネル扱いなど、ブラウザ差で壊れる可能性がある箇所がある
- Prism/Noise が「生成のみ」でデコード手段がページ内に無い（v1.0でどう扱うか決める）
- Animated Prism は“完成”主張と実装がズレている可能性がある
- ページ間リンクに存在しないHTML参照がある可能性（リンク切れ）

## 6. フェーズ計画（Phase 1〜10）
下表の順に進める。各Phaseは **小さくコミット**し、受け入れ条件を満たしたら次へ。

| Phase | 目的 | 主な成果物 | 受け入れ条件（抜粋） |
|------:|------|------------|-----------------------|
| 1 | ドキュメントで“完成定義”を固定し、READMEと実装の整合を取る | docs一式、README整合 | 未実装主張ゼロ / 目的と限界が明記 |
| 2 | Stego Snap安定化（イベント依存排除・LSBの整合） | index.html修正 | roundtrip 20回連続成功（PNG） |
| 3 | Prism Codeにデコード機能追加（ページ内完結） | prism.html拡張 | 生成→保存→再読込→デコード成功 |
| 4 | Noise Talkにデコード機能追加（ページ内完結） | noise.html拡張 | 生成→保存→再読込→デコード成功 |
| 5 | Multi-Layerのリンク/UX/堅牢性を磨く | multi-layer-stego.html改善 | リンク切れゼロ / decode結果の明確化 |
| 6 | 共通ロジックのモジュール化（重複削減） | src/導入、HTMLから利用 | 重複ロジック削減・可読性向上 |
| 7 | 自動テスト＆CI導入 | GitHub Actions / unit tests | PRでテストが回り、破壊を検知 |
| 8 | Animated Prism “MVP”を実装して実在化 | animated-prism.html刷新 | encode/decodeが成立（規定条件） |
| 9 | Animated Prismの耐性/性能を強化 | 冗長化/多数決/変形/最適化 | 成功率・クラッシュ率の改善 |
| 10 | v1.0リリース作法（仕上げ） | CHANGELOG/タグ/テンプレ | v1.0合格条件を満たしタグ付け |

## 7. リリース方針
- 破壊的変更はフォーマットをバージョン化して回避する
- 実装と説明がズレた瞬間に信用が死ぬので、READMEは常に実装に従属
